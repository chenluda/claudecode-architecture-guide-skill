## 第二部分：通用迁移策略

### 策略 1：Strangler Fig 渐进替换

**何时使用**：改造现有系统，不能停机重写。

```
阶段 1：新旧并存
┌─────────────────────────┐
│  新接口（包装旧实现）      │
│  ┌───────────────────┐  │
│  │  旧实现（黑箱）      │  │
│  └───────────────────┘  │
└─────────────────────────┘

阶段 2：逐步替换
┌─────────────────────────┐
│  新接口                   │
│  ┌────┐ ┌────────────┐  │
│  │ 新  │ │  旧（缩小） │  │
│  └────┘ └────────────┘  │
└─────────────────────────┘

阶段 3：完全替换
┌─────────────────────────┐
│  新接口（新实现）          │
└─────────────────────────┘
```

**步骤**：
1. 定义新接口（不改旧实现）
2. 用新接口包装旧实现（适配器模式）
3. 所有调用方迁移到新接口
4. 逐个用新实现替换旧实现
5. Feature flag 控制新旧切换

### 策略 2：Feature Flag 灰度迁移

**何时使用**：高风险变更需要逐步验证。

```
Phase 1: flag=false for all         → 100% 走旧路径
Phase 2: flag=true for 5% (canary)  → 小流量验证
Phase 3: flag=true for 50%          → 扩大验证
Phase 4: flag=true for 100%         → 全量上线
Phase 5: remove flag + old code     → 清理
```

**关键**：
- 运行时 flag 控制路径选择
- 两条路径必须结果等价（可对比验证）
- 有回滚方案（flag 切回 false）
- Phase 5 不可省略（技术债清理）

### 策略 3：配置驱动迁移

**何时使用**：改变行为但不改变代码路径。

```typescript
// 迁移前
const RETRY_LIMIT = 3

// 迁移后
const config = buildConfig()  // 从配置/远程读取
const RETRY_LIMIT = config.retryLimit ?? 3  // 有默认值
```

**步骤**：
1. 识别硬编码值
2. 定义配置 schema（Zod）
3. 添加默认值（向后兼容）
4. 引入配置源层次（CLI > 项目 > 用户 > 默认）
5. 添加迁移脚本（处理格式变更）

## 第四部分：常见陷阱

### 陷阱 1：过早平台化

**症状**：0→1 阶段就建设插件市场和 Hook 系统。

**药方**：先用硬编码验证核心价值，再抽象。Claude Code 的插件系统 `initBuiltinPlugins()` 当前为空实现 — 先占位，后填充。

### 陷阱 2：大爆炸迁移

**症状**：一次性替换整个子系统。

**药方**：Strangler Fig + Feature Flag。Claude Code 的 `feature('REACTIVE_COMPACT')` 允许新旧压缩策略并存。

### 陷阱 3：忽视恢复路径

**症状**：只测试 happy path，错误直接抛出。

**药方**：为每种失败场景设计恢复链。Claude Code 的 `maxOutputTokensRecoveryCount` 有明确上限（3 次），超限才真正报错。

### 陷阱 4：全局状态蔓延

**症状**：越来越多的模块直接读写全局变量。

**药方**：Claude Code 在 `bootstrap/state.ts` 顶部注释"DO NOT add more global state without careful consideration"。用分层状态替代全局状态。

### 陷阱 5：权限检查遗漏

**症状**：新增工具/命令忘记添加权限检查。

**药方**：权限检查嵌入工具注册流程（`CanUseToolFn` 作为必传参数），而非靠开发者记住。

### 陷阱 6：遥测后补

**症状**：线上问题无法定位，才开始加日志。

**药方**：Claude Code 的 `profileCheckpoint()` 从 `main_function_start` 就开始埋点。遥测是架构的一部分，不是事后补丁。

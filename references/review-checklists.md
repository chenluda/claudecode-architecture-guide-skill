# 架构评审清单

基于 Claude Code 工程实践提炼的评审清单。适用于架构设计评审、代码审查、上线检查等场景。

---

## 清单 A：核心架构评审

### A1. 分层与边界

- [ ] 系统有明确的层次划分（编排 / 执行 / 策略 / 服务 / 存储）
- [ ] 每层通过接口（而非具体实现）与其他层交互
- [ ] 编排层不直接处理重试、权限、序列化等细节
- [ ] 策略层可通过配置修改，无需改代码
- [ ] 跨层依赖方向单一（高层依赖低层，不反向）

### A2. 状态管理

- [ ] 可变状态集中定义（有明确的 State 类型）
- [ ] 不可变配置在入口点快照（不在循环中重读）
- [ ] 全局状态有显式的 getter/setter（非直接暴露变量）
- [ ] 状态的可变范围 ≤ 其生命周期（不跨层持有可变引用）
- [ ] 状态变更有变更通知机制（subscribe/onChange）

### A3. 错误处理

- [ ] 有自定义错误类型层次（不全用 Error）
- [ ] 可恢复错误与不可恢复错误明确区分
- [ ] 恢复策略有明确的重试上限
- [ ] 错误不会破坏协议配对（如 request/response、tool_use/tool_result）
- [ ] 有优雅关停机制（cleanup 函数、signal 监听）

### A4. 并发模型

- [ ] 并发安全性在操作定义侧声明（非调度侧判断）
- [ ] 读写操作有明确的并发策略（只读可并发 / 写入串行）
- [ ] 有取消传播机制（abort controller 或类似）
- [ ] 并发度可配置（非硬编码）
- [ ] 失败不会导致未清理的孤儿操作

---

## 清单 B：安全与权限评审

### B1. 权限模型

- [ ] 每个外部操作都经过权限检查（无绕过路径）
- [ ] 权限决策结果是类型化的（allow/deny/ask，非 boolean）
- [ ] 拒绝有明确的原因记录（可审计）
- [ ] 权限规则来源可追溯（管理员/用户/项目配置）
- [ ] 权限模式支持多种场景（交互/无头/自动）

### B2. 信任边界

- [ ] 外部输入（用户、网络、插件）有验证层
- [ ] Schema 验证在入口处完成（Zod / JSON Schema）
- [ ] 插件/扩展有分级信任（内置 > 官方市场 > 第三方 > 临时）
- [ ] 网络访问有 SSRF 防护
- [ ] 敏感操作有确认机制

### B3. 沙箱与隔离

- [ ] 危险操作（shell、文件写入）有沙箱选项
- [ ] 沙箱策略可与权限系统联动（沙箱内的操作可自动放行）
- [ ] 子进程有独立的 abort controller
- [ ] 临时资源在会话结束时清理

---

## 清单 C：性能评审

### C1. 启动性能

- [ ] 关键路径 I/O 并行预取（不串行等待）
- [ ] 重模块延迟加载（OpenTelemetry、分析、非核心功能）
- [ ] 初始化有 memoize 保证（不重复执行）
- [ ] 有启动性能 checkpoint 测量各阶段耗时
- [ ] 未使用的功能通过 feature flag 从 bundle 中消除

### C2. 运行时性能

- [ ] 流式处理使用 AsyncGenerator（非回调累积）
- [ ] 大结果有预算控制（截断/摘要）
- [ ] 状态变更有 bailout 机制（Object.is 或深度比较）
- [ ] 工具排序稳定（利于 prompt cache）
- [ ] 长操作有进度反馈机制

### C3. 内存管理

- [ ] 流式数据不在内存中全量累积
- [ ] 上下文有压缩/裁剪策略（auto compact / snip / collapse）
- [ ] 订阅有取消机制（防止 listener 泄漏）
- [ ] 使用 `using` / `try-finally` 确保资源清理

---

## 清单 D：可扩展性评审

### D1. 工具/命令系统

- [ ] 新工具/命令可通过标准接口添加（不修改核心代码）
- [ ] 每个工具自包含：schema、权限、执行逻辑、UI
- [ ] 工具注册有条件过滤（环境、模式、权限规则）
- [ ] MCP/外部工具与内置工具在同一接口下合并
- [ ] 工具列表排序稳定且可配置

### D2. Hook/插件系统

- [ ] 生命周期事件覆盖完整（pre/post/error 每个关键点）
- [ ] Hook 是语言无关的（JSON 协议，非 API 绑定）
- [ ] Hook 可阻塞执行流（blocking 模式）
- [ ] 插件目录结构有明确约定（commands/ / agents/ / hooks/）
- [ ] 插件有启用/禁用管理

### D3. 配置与策略

- [ ] 配置有 schema 验证（Zod / JSON Schema）
- [ ] 配置来源分层（CLI > 项目 > 用户 > 默认）
- [ ] 配置变更有迁移机制（版本化 migration）
- [ ] 远程策略可覆盖本地配置（企业管控）
- [ ] 配置加载失败有优雅降级（不阻断启动）

---

## 清单 E：可观测性评审

### E1. 遥测

- [ ] 关键操作有性能 checkpoint
- [ ] 有结构化的分析事件（非 console.log）
- [ ] 遥测初始化延迟加载（不增加启动开销）
- [ ] 有费用/token 追踪（成本可见）
- [ ] 会话有唯一 ID 可追踪

### E2. 诊断

- [ ] 错误有分类标签（可聚合分析）
- [ ] 有 doctor 命令/诊断工具（环境自检）
- [ ] 有调试日志层（可选开启，不影响生产）
- [ ] 有 transcript/会话记录（可回放分析）

---

## 清单 F：Agent/LLM 特化评审

### F1. 对话管理

- [ ] 多轮对话有显式的消息类型系统（不是裸字符串）
- [ ] tool_use 和 tool_result 总是配对（包括异常场景）
- [ ] 上下文窗口有分层预算管理（而非等 prompt_too_long）
- [ ] 有自动压缩策略（auto/reactive/micro）
- [ ] 思维链（thinking）块有明确的保留与清理规则

### F2. 模型调用

- [ ] 流式响应有完整的事件类型覆盖
- [ ] 有模型降级/fallback 机制
- [ ] 重试有分类（可重试错误 vs 不可重试错误）
- [ ] 有 token 预算追踪和上限控制
- [ ] 模型选择可运行时切换

### F3. 子 Agent 协调

- [ ] 子 Agent 有独立的消息上下文（不污染主会话）
- [ ] 子 Agent 有独立的 abort controller（可独立取消）
- [ ] 有 transcript/侧链记录
- [ ] 权限决策可委托给协调器
- [ ] 有最大深度/并发限制

---

## 使用建议

1. **新项目**：重点关注清单 A、B — 确保基础架构正确
2. **增长期项目**：追加清单 C、D — 性能和扩展性
3. **平台化项目**：全部清单 — 特别是 E、F
4. **Agent/LLM 项目**：必须完成清单 F，结合 A、B

每个检查项不是"必须全部勾选"，而是"逐项审议，有意识地决定是否需要"。跳过某项应有明确理由记录。
